# バックアップ / リストア運用手順

## 目的

本手順は `winningpost10-manager` のローカル SQLite データを定期的に退避し、障害時に安全に復旧するための運用標準です。

## バックアップ（定期運用）

1. アプリを起動し、`設定` > `データベース` セクションを開く。
2. `エクスポート` を実行する。
3. `wp10-manager-backup-YYYYMMDD-HHmmss.sqlite3` 形式のファイルがダウンロードされることを確認する。
4. 取得ファイルを以下のいずれかに保管する。
   - 端末内のバックアップ専用フォルダ
   - バージョン管理外の共有ストレージ
   - 暗号化されたクラウドストレージ

### 推奨頻度

- 通常運用: 週1回以上
- 大量データ更新前後: 作業直前・作業直後
- アプリ更新前: 必ず1回

## 障害時リストア

1. 事前に直近の正常バックアップを準備する。
2. `設定` > `データベース` で `データベースリストア` をクリックし、対象ファイルを選択する。
3. 1回目確認ダイアログ（ファイル名表示）で内容を確認し `次へ`。
4. 2回目確認ダイアログ（最終確認）で `リストア実行`。
5. 完了後、アプリが再読込され、UI状態とDB状態が同期されることを確認する。
6. バージョン不一致エラー時は、同一アプリバージョンで取得したバックアップを選択して再試行する。

## 検証手順（復旧確認）

リストア実行後、最低限以下を確認する。

- 年度設定（`現在の年度`）が期待値である。
- 主要データ（繁殖牝馬 / 系統 / 配合理歴）が件数・内容ともに想定通り。
- 設定画面の `バージョン` が期待するDBバージョンである。
- 必要に応じてテストデータ投入が正常動作する。

## 失敗時の挙動（ロールバック方針）

- リストア処理はトランザクション内で実行される。
- リストア中に例外が発生した場合、トランザクションをロールバックし既存DBを維持する。
- 失敗時は画面にエラーメッセージを表示し、再実行前にバックアップファイルの正当性を確認する。

## 注意点（互換性・バージョン差異）

- バックアップファイルはJSONダンプで、作成日時・スキーマバージョンをメタ情報として含む。
- 将来のスキーマ変更により、旧バックアップの取り込み可否が変化する可能性がある。
- アプリ更新時は、更新前バックアップを保持し、更新後すぐに復元テストを実施する。
- 重大なバージョン差異がある場合は検証環境で事前リハーサルを行う。
